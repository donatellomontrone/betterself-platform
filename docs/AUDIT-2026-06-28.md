# BetterSelf Audit Status

Original audit date: 2026-06-28
Status refresh: 2026-07-04

This file replaces the stale long-form audit with the current remediation state.
The original audit found 54 issues across payment flow, auth/data handling,
privacy, accessibility, SEO, and UX. The codebase has since been remediated
through several hardening passes.

## Current status

- Known P0 issues: none in code.
- Original P1 issues: remediated in code.
- Production risk now sits mostly in external account readiness: PayMongo live
  activation, webhook delivery, legal details, and final live checkout smoke test.

## Remediated P1 findings

1. Unit/area treatment undercharge
   - Added doctor-assessed amount support for treatments whose final price depends
     on units or areas.
   - Dashboard Pay now is gated until the doctor confirms the booking and, when
     needed, records the final assessed amount.
   - Retry checkout now uses the confirmed amount before falling back to catalog
     starting price.

2. PayMongo webhook payload mismatch
   - Webhook now supports both the event-wrapper payload shape and the direct
     `checkout_session.payment.paid` shape.
   - Reference extraction handles `data.attributes.data.attributes.reference_number`
     and direct resource attributes.
   - HMAC signature verification remains fail-closed.

3. Cancelled booking resurrected by late payment
   - `markPaidByReference` no longer updates cancelled bookings to paid.
   - Captured payments against cancelled bookings are logged for manual
     reconciliation/refund.
   - Paid consultations are advanced to confirmed where appropriate.

## Other important remediation completed

- Consultation and treatment payment flows were separated:
  - consultation: pay first, then schedule the call;
  - treatment: doctor review first, then payment from dashboard.
- PayMongo retry creates a new QR Ph checkout session, avoiding "source consumed"
  dead ends.
- Demo retry mode no longer loops forever on `/checkout-preview`.
- Discount codes are validated server-side.
- Admin CSV export is protected against spreadsheet formula injection.
- Admin CSV export logs who exported and how many rows were exported.
- Admin partial profile edits preserve existing medical fields instead of wiping
  them with blank values.
- OpenAI recommendation input is capped at 1000 characters before any paid API
  call.
- Recommendation output is schema-constrained and has a safe local fallback.
- Cookie banner now gates optional Google Places loading.
- SEO metadata, robots, sitemap, Open Graph, and noindex behavior were added.
- Clerk CSV/proxy matcher was fixed so protected CSV export still runs through
  Clerk middleware.
- Admin dashboard gained richer operational views: booking filters, patient view,
  payment queue, calendar sections, doctor-assessed amount, and message context.

## Remaining verification before real patients

These are not primarily code fixes, but should be verified before accepting live
payments/patients.

1. PayMongo live smoke test
   - Confirm merchant/business verification and QR Ph channel activation.
   - Set live `PAYMONGO_SECRET_KEY` and `PAYMONGO_WEBHOOK_SECRET` in Vercel.
   - Register webhook:
     `https://betterself.health/api/paymongo/webhook`
   - Send a test/live checkout event and confirm the booking becomes paid in Neon.

2. Clerk production signup test
   - Sign up in an incognito browser with a fresh email.
   - Confirm redirect to `/dashboard`.
   - Confirm user profile and patient profile data appear in Neon/admin after a
     booking attempt.

3. Legal/content review
   - Fill legal entity, registered address, DPO/contact, refund/cancellation terms,
     and medical operator details where needed.
   - Have the doctor/lawyer approve Privacy, Terms, and Consent.

4. Clinical content
   - Replace generic treatment copy with doctor-approved mechanism, indications,
     aftercare, downtime, contraindication, and expectation text per treatment.

5. Optional DB migration
   - Consider a deliberate backup + migration for consent/account-consent
     deduplication if duplicate historical rows exist.

## Audit posture

The code is much closer to a production-ready MVP than the original audit
suggested. The remaining risks are operational: live payment activation, legal
accuracy, and real-world clinical process validation.
